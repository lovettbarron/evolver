### Pattern: Cable ID Resolution (the core bug)

**Current broken code** (patch-detail.tsx lines 127-131):
```typescript
// Transforms "VCO A SAW OUT" -> "jack-vco-a-saw-out"
sourceId: `jack-${c.source.toLowerCase().replace(/\s+/g, '-')}`,
```

**Problem:** Cascadia CONTROL_METADATA keys include the full module prefix. Example:
- Patch YAML: `source: "LFO X OUT"` -- transforms to `jack-lfo-x-out`
- Actual key: `jack-lfo-xyz-x-out` (module is `lfo-xyz`, not `lfo`)
- Another: `source: "Envelope A ENV OUT"` -- transforms to `jack-envelope-a-env-out`
- Actual key: `jack-envelope-a-out` (display name is "ENV A", not "Envelope A ENV OUT")

**Fix approach:** Build a reverse lookup map from CONTROL_METADATA `name` field to control `id`. The `name` field stores the silk-screen label (e.g., "LFO X", "ENV A"). The patch YAML `source`/`destination` strings are human-readable descriptions that need matching to these names. A lookup function should:
1. Build a map from normalized `name` -> `id` for all jacks in CONTROL_METADATA
2. Handle the "OUT"/"IN" suffix stripping (display names don't include directional suffixes)
3. Fall back to the current kebab transform if no match found

### Pattern: SignalType Resolution

**Current:** `signalType: 'default' as const` hardcoded for all cables.

**Fix:** Each jack in CONTROL_METADATA has a `signalType` field ('audio' | 'cv' | 'gate' | 'modulation'). After resolving the source jack ID, look up its signalType from CONTROL_METADATA. Use the source jack's signalType for the cable color (source determines signal type).

### Pattern: Section Tint Opacity

## Common Pitfalls

### Pitfall 1: Cable Name Matching Ambiguity
**What goes wrong:** Multiple jacks could match a partial name (e.g., "VCA" matches VCA A and VCA B)
**Why it happens:** Patch YAML uses free-text descriptions, not control IDs
**How to avoid:** Match against the full display name with module context. Build lookup from both `name` and `module+name` combinations. Test with all 13 existing Cascadia patches.
**Warning signs:** Cables rendering to wrong jacks or not rendering at all

### Pitfall 2: Opacity Value Cascading
**What goes wrong:** Fixing the test but not the implementation (or vice versa) leaves a silent inconsistency
**How to avoid:** Decide on one value, update both implementation files (evolver-panel.tsx, cascadia-panel.tsx) AND both test files, verify with `npx vitest run`
**Warning signs:** Tests pass but panels look visually different between instruments

### Pitfall 3: Orphaned Import Cleanup
**What goes wrong:** Deleting session-panel-sidebar.tsx but leaving imports elsewhere
**Why it happens:** File might be imported in places grep didn't catch
**How to avoid:** Search confirmed zero imports (grep found no matches). Safe to delete.

## Code Examples

### Reverse Lookup for Cable IDs
```typescript
// Build reverse lookup: normalized display name -> jack control ID
// Source: cascadia-panel-data.ts CONTROL_METADATA
import { CONTROL_METADATA } from '@/lib/cascadia-panel-data';

function buildJackLookup(): Map<string, string> {
  const lookup = new Map<string, string>();
  for (const [id, meta] of Object.entries(CONTROL_METADATA)) {
    if (meta.type === 'jack-in' || meta.type === 'jack-out') {
      // Map display name to ID (e.g., "LFO X" -> "jack-lfo-xyz-x-out")
      lookup.set(meta.name.toLowerCase(), id);
    }
  }
  return lookup;
}

// In patch-detail.tsx cable mapping:
const jackLookup = buildJackLookup();
const sourceId = jackLookup.get(normalizeLabel(c.source))
  ?? `jack-${c.source.toLowerCase().replace(/\s+/g, '-')}`;
```
