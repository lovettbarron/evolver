The global :focus-visible would add a conflicting outer ring. Suppress it:
```css

}
```

NOTE: This requires search-bar.tsx to add the class `search-input` to its input element. That will happen in Plan 02 Task 1.

Also suppress the focus outline on elements that already define their own focus treatment via `focus:outline-none` + `focus:ring-*` patterns:
```css
/* Elements with custom focus rings suppress the global outline */
[class*="focus:ring"]:focus-visible,
[class*="focus:outline-none"]:focus-visible {
  outline: none;
}
```

Wait -- that won't work with Tailwind class matching at CSS level. Instead, just add the `.search-input` suppression. The other elements (quick-ref-panel button, sticky-header button) use `focus:ring-2 focus:ring-accent` which Tailwind generates as `outline: none` + ring. The global `:focus-visible` and Tailwind's `focus:outline-none` utility will interact -- Tailwind's utility-layer `outline-none` will be overridden by the unlayered global `:focus-visible`. To handle this properly, also add:

```css
/* Patch-save-form inputs with custom focus borders */
.prose-form-input:focus-visible {
  outline: none;
}
```

Actually, the simplest and most correct approach: use `:where()` on the global rule to give it zero specificity, so any component-level `outline-none` or `focus:outline-none` Tailwind utility beats it:

```css
/* Global keyboard focus indicator (D-05) — :where() for zero specificity so component overrides win */
:where(:focus-visible) {
  outline: 2px solid var(--color-accent);
  outline-offset: 2px;
}
```

This way:
- All interactive elements get the accent outline by default (keyboard only)
- Components that already use `outline-none` or `focus:outline-none` (search input, patch-save-form, quick-ref button, sticky-header button) naturally override it because Tailwind utilities have higher specificity than `:where()` selectors
- No per-component suppressions needed

Use `:where(:focus-visible)` instead of bare `:focus-visible`.
